Circuit Breaker Lock Mechanism

A sophisticated pattern for graceful degradation and automatic recovery in distributed systems

Modern distributed systems require robust failure handling mechanisms.
Learn how circuit breaker patterns can protect your system and ensure message integrity.

Architecture Overview

Multi-process stream monitoring system with coordinated circuit breaker pattern using Worker orchestration, instrumentation hooks, and comprehensive error handling.

🔒 Lock Events & Instrumentation

kafka_lock_event

Controls Kafka message consumption + instrumentation.lock_set("kafka")

data_api_lock_event

Controls queue and cache processing + instrumentation.lock_set("data_api")

Individual Lock Control

set_kafka_lock(), clear_data_api_lock(), with granular control

⚙️ Worker Process Architecture

Kafka Consumer Worker

Async process with topic-refresh loop, rebalance listener, cooperative sleep

Queue Consumer Worker

Sync process with MessageManager, GRPC client, retry handler

Cache Consumer Worker

Async process reading cached messages, clearing kafka locks on empty

ServiceManager

Orchestrates Worker lifecycle with Event coordination

🔄 Key Components

ErrorHandler: Type-based dispatch (JSON, KeyError, ValueError) → Dead Letter
MessageFilterService: Split batches by IgnoreDuplicate flag
GRPCClientManager: Channel readiness + MessageBuilderRegistry
Instrumentation: Metrics hooks throughout (lock_set, message_rate, process_status)

Lock Types & Components

🚦 LockManager API & Instrumentation

Individual Lock Controls

set_kafka_lock(): Stops Kafka consumption

set_data_api_lock(): Stops queue/cache processing

clear_kafka_lock(): Resumes Kafka consumption

clear_data_api_lock(): Resumes queue processing

Coordinated Lock Controls

set_all_locks(): Emergency stop - both systems

clear_all_locks(): Recovery - clear data_api first, then kafka

Instrumentation: Every lock operation calls instrumentation.lock_set() or lock_clear()

class LockManager: def set_kafka_lock(self) -> None: self.kafka_lock_event.set() instrumentation.lock_set("kafka") def clear_all_locks(self) -> None: self.clear_data_api_lock() # Clear data API first self.clear_kafka_lock() # Then clear kafka instrumentation.lock_clear("kafka") instrumentation.lock_clear("data_api")

Process Coordination & Worker Orchestration

Kafka Consumer Worker

Async Process: topic-refresh loop, ConsumerRebalanceLogger, cooperative_sleep()

Lock Response: kafka_lock_event.is_set() → sleep(0.3)

Queue Consumer Worker

Sync Process: MessageManager, GRPCOperationHandler, RetryHandler

Processing Mode: ThreadPoolExecutor (parallel) when shared_worker_threading_flag=True, else sequential

Lock Response: Sets locks on GRPC failures, clears on success

Cache Consumer Worker

Async Process: get_message_from_cache(), clears kafka_lock on empty cache

Lock Response: data_api_lock_event.is_set() → skip queue insertion

Lock Behavior & GRPC Integration

⚡ Lock Triggering Scenarios

Kafka Lock Activation:
  • GRPC Retry (on_retry): set_kafka_lock() during retries
  • Queue Backpressure: queue.qsize() >= shared_queue_enqueue_size
  • Cache Empty: kafka_lock_event.clear() when cache drained
Data API Lock Activation:
  • Server Unavailable: GrpcServerNotConnectingException
  • Retry Exhaustion (on_final_failure): set_all_locks()
  • Queue Empty: data_api_lock_event.clear() when queue drained

Error Handling & Recovery

🔧 Multi-Layer Error Handling

ErrorHandler Type Dispatch

json.JSONDecodeError: Malformed message → Dead Letter

KeyError: Missing required fields → Dead Letter

ValueError/TypeError: Data validation → Dead Letter

Generic Exception: Fallback → Dead Letter

GRPC Error Handling

ALREADY_EXISTS: Check IgnoreDuplicate flag

UNAVAILABLE/DEADLINE_EXCEEDED: Retryable → Cache + Retry

Other StatusCodes: Non-retryable → Dead Letter

🔄 Cache Reading & Recovery Flow

Cache Consumer Process

get_message_from_cache(): JSON decode with EnumDecoder

consume_cache_messages(): Batch processing up to cache_batch_size

Empty Cache: kafka_lock_event.clear() to resume consumption

Dead Letter Production

Topic: shared_kafka_dead_letter_topic

Payload: {msg, grpc_api_call_name, request, error}

Instrumentation: message_rate, delivery callbacks

# ErrorHandler Type-Based Dispatch class ErrorHandler: def __init__(self, dead_letter_producer): self.error_map = { json.JSONDecodeError: self._handle_json_error, KeyError: self._handle_key_error, ValueError: self._handle_value_error, TypeError: self._handle_type_error, } def handle_error(self, error, message, grpc_call, request_json): handler = self.error_map.get(type(error), self._handle_generic_error) handler(error, message, grpc_call, request_json) # Cache Recovery with Lock Clearing async def consume_cache_messages(queue, event, kafka_lock_event): while event.is_set(): message = get_message_from_cache() if message: messages.append(message) else: if kafka_lock_event.is_set(): kafka_lock_event.clear() # Resume Kafka consumption break

Failure Scenario Walkthrough

Step-by-step analysis of how the circuit breaker handles a typical Data API outage with actual implementation details

📚 Scenario: Data API Goes Down During Peak Load

Initial State (T=0s):
  • Kafka producing 1000 msg/sec → instrumentation.message_rate("kafka_consumed", n)
  • Queue has 500 messages → instrumentation.queue_update(size, "message_added")
  • GRPCClientManager.is_server_available() → True
  • All locks cleared → instrumentation.lock_clear("kafka"), lock_clear("data_api")
Data API Failure (T=5s):
  • GRPC calls → StatusCode.UNAVAILABLE
  • Queue consumer detects channel_ready_future timeout
  • RetryHandler.execute_with_retry() activates
  • on_retry callback → lock_manager.set_kafka_lock() + instrumentation.lock_set("kafka")

⚡ Timeline: First 30 Seconds

T=5s - GrpcServerNotConnectingException T=6s - Retry 1/5 → set_kafka_lock() + instrumentation T=8s - Retry 2/5 (backoff: 2s) → cooperative_sleep_sync() T=12s - Retry 3/5 (backoff: 4s) T=20s - Retry 4/5 (backoff: 8s) T=33s - on_final_failure triggered → set_all_locks() + instrumentation.lock_set("data_api") → _cache_raw(message) with EnumEncoder → Queue processing STOPPED

🔄 Recovery Sequence

T=120s - Data API channel_ready_future succeeds T=125s - Cache consumer: get_message_from_cache() → consume_cache_messages() batching → Server availability check succeeds → clear_all_locks() (data_api first, then kafka) T=126s - kafka_lock_event.clear() → instrumentation.lock_clear("kafka") T=127s - Cached messages → queue → MessageFilterService T=130s - Normal operation → instrumentation.message_rate("queue_processed", n)

🎯 Key Implementation Details

Message Preservation: During outage, ~120,000 messages handled via 3 paths:
• Kafka Queue: Buffered due to kafka_lock_event.is_set() check
• Cache Storage: Failed messages → _cache_raw() with JSON+EnumEncoder
• Dead Letter: Non-retryable errors → DeadLetterProducer.produce_message()
Zero Data Loss: ErrorHandler type dispatch + MessageFilterService IgnoreDuplicate handling

Configuration Tuning Guide

Critical parameters and recommended values for different deployment scenarios

⚙️ Retry Configuration

MAX_RETRIES

Default: 5 attempts

Low latency env: 3 (faster failure detection)

High latency env: 7 (more tolerance)

INITIAL_BACKOFF_S

Default: 1.0s

High throughput: 0.5s (faster recovery)

Resource constrained: 2.0s (less aggressive)

RPC_DEADLINE_S

Default: 30s

Fast networks: 10s

Unreliable networks: 60s

🚦 Queue & Memory Management

shared_queue_enqueue_size

Default: 1000 messages

High memory systems: 5000

Memory constrained: 500

data_api_channel_ready_timeout

Default: 5s

Fast detection: 2s

Stable networks: 10s

kafka_consume_interval_sec

Default: 0.1s

High throughput: 0.05s

Troubleshooting Common Issues

Diagnostic steps and solutions for typical circuit breaker problems with actual implementation context

🚨 Issue #1: Individual Locks Not Clearing

Symptoms

kafka_lock_event.is_set() = True but queue is empty, or data_api_lock_event stuck

Root Cause

Cache consumer Worker crashed before kafka_lock_event.clear() or queue consumer died

Diagnosis
# Check individual lock states print(f"Kafka lock: {kafka_lock_event.is_set()}") print(f"Data API lock: {data_api_lock_event.is_set()}") # Check Worker process status for worker in [kafka_worker, queue_worker, cache_worker]: print(f"{worker.name}: alive={worker.is_alive()}, event_set={worker.event.is_set()}") # Manual recovery lock_manager.clear_all_locks()

⚡ Issue #2: Worker Process Crashes

Symptoms

instrumentation.process_status shows "inactive", no logs from specific worker

Root Cause

Exception in Worker.run(), asyncio loop issues, or os._exit(0) called

Solution
# Check process health via ServiceManager if not worker.is_alive(): logger.error(f"Worker {worker.name} died, restarting") worker.cleanup() worker = Worker(queue, task, args, worker_type=worker_type) worker.start() # Check event loop state for async workers if worker.loop and worker.loop.is_closed(): logger.warning("Event loop closed unexpectedly")

🔍 Issue #3: Memory Leak Prevention

Symptoms

RAM usage growing, queue.qsize() approaching system limits

Root Cause

Queue growth exceeding shared_queue_enqueue_size, no backpressure handling

Prevention (Built-in)
# Built-in queue size monitoring in consume_internal_queue_messages if queue.qsize() >= shared_queue_enqueue_size.value: logger.debug(f"Queue size {current_queue_size} >= {shared_queue_enqueue_size.value}") await cooperative_sleep(event, interval_sec) # Backpressure # Emergency: Drop oldest pattern (if implemented) if queue.qsize() > MAX_QUEUE_SIZE: try: queue.get_nowait() # Drop oldest message except Empty: pass

⏱️ Issue #4: Cache Check Timeouts

Symptoms

Cache consumer sleeping too long, slow recovery on cache drain

Root Cause

shared_cache_check_empty_timeout_sec too high, blocking shutdown signals

Solution (Already Implemented)
# Cooperative sleep breaks long timeouts into chunks await cooperative_sleep( event, shared_cache_check_empty_timeout_sec.value, step=1.0 # 1-second responsiveness ) # Queue consumer similar pattern sleep_chunk = 0.1 # 100ms granularity for _ in range(int(shared_queue_check_empty_timeout_sec.value / sleep_chunk)): if not event.is_set(): return # Responsive to shutdown cooperative_sleep_sync(event, sleep_chunk, step=1)

🎯 Issue #5: Parallel Processing Bottlenecks

Symptoms

ThreadPoolExecutor overwhelmed, shared_worker_threading_flag=True but slow processing

Root Cause

shared_worker_max_thread_count vs shared_worker_max_rpc_batch_size mismatch

Solution
# process_messages_batch with ThreadPoolExecutor if shared_worker_threading_flag.value: batch_size = shared_worker_max_rpc_batch_size.value with ThreadPoolExecutor( max_workers=shared_worker_max_thread_count.value ) as executor: # Chunk into sub-batches for batch_start in range(0, len(messages), batch_size): # Submit to thread pool executor.submit(process_func, ...)

⚡ Issue #6: Dead Letter Storms

Symptoms

shared_kafka_dead_letter_topic overwhelmed, ErrorHandler._handle_*_error logs

Root Cause

Malformed messages causing JSON/KeyError/ValueError in bulk

Prevention
# Add message validation before ErrorHandler try: if "message" not in message: raise KeyError("Bad Message key. Key 'message' not found") parsed = orjson.loads(message["message"]) except (json.JSONDecodeError, KeyError) as e: error_handler.handle_error(e, message, grpc_call, request_json) return # Don't process further
